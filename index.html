A guide for developers

From requirements doc to running code with a team of AI agents

Use Claude Code to review requirements, generate an implementation plan, and then send a team of agents to build it. You stay in the driver's seat — the agents do the heavy lifting.

The workflow has three phases. First, you give Claude a requirements document and ask it to review and plan — it will break the work into tasks and produce a plan file. Second, you read and approve the plan, making changes as needed. Third, you tell Claude to implement the plan, and it dispatches a team of subagents to build in parallel.

You're not handing off control. You're directing a team — reviewing their plan before they write a line of code, and staying available to course-correct as they work.

Setup

If you already have Claude Code installed and working, skip to the workflow section.

1

Install Claude Code

Terminal $ curl -fsSL https://claude.ai/install.sh | bash

Verify it works:

Terminal $ claude --version

The first time you run Claude, it will ask you to log in. Follow the prompts — it opens a browser window.

2

Prepare your requirements

Export your requirements document as Markdown. If it's in Google Docs, use an add-on like "Docs to Markdown" or just copy-paste the content into a .md file. Save it somewhere accessible — you'll point Claude at it.

Example file structure # Your project folder ~/Projects/my-app/ requirements.md # The requirements doc
What makes a good requirements doc? Clear user stories, acceptance criteria, and a list of screens or features. It doesn't need to be perfect — Claude will ask clarifying questions. But the more specific you are about what you want, the better the plan will be.

The review → plan → build workflow

3

Start Claude in your project directory

Navigate to your project folder and start Claude:

Terminal $ cd ~/Projects/my-app $ claude
4

Ask Claude to review requirements and create a plan

Point Claude at your requirements file and ask it to analyze and plan. Be explicit about your tech stack preferences:

You → Claude Read requirements.md. Review the requirements thoroughly. Ask me any clarifying questions before proceeding. Then create a detailed implementation plan in PLAN.md that breaks the work into concrete tasks. Each task should be small enough for one agent to handle independently. Tech stack: - Next.js with TypeScript - Tailwind CSS + shadcn/ui for the frontend - Supabase for the database and auth - Supabase Storage (S3 compatible) for file uploads - Include a README.md explaining how to run locally

Claude will read the requirements, likely ask you a few questions, and then produce a PLAN.md file with a structured breakdown of the work.

Let Claude ask questions first Good plans come from good understanding. Don't rush past Claude's clarifying questions — they usually surface ambiguities you hadn't noticed. Answer them, then let it finalize the plan.
5

Review the plan yourself

This is the most important step. Open PLAN.md in your editor and read it carefully. Look for:

  • Does the task breakdown make sense? Are tasks appropriately scoped?
  • Are there dependencies between tasks that need ordering?
  • Does it match your understanding of the requirements?
  • Is anything missing or over-engineered?
  • Does the tech stack match what you specified?

Edit the plan directly if you want to change things. Add notes, remove tasks, reorder priorities. This is your chance to steer before any code is written.

Don't skip the review It's tempting to say "looks good, go build it." Resist. Spending 10 minutes reviewing the plan saves hours of rework. The plan is your contract with the agents — make sure it says what you want.
6

Send the agents to build

Once you're happy with the plan, tell Claude to implement it. Claude will dispatch subagents to work on tasks in parallel:

You → Claude I've reviewed PLAN.md and it looks good. Implement the plan. Use subagents to work on independent tasks in parallel. Create a git commit after each major milestone. Make sure the README.md has clear setup instructions.

Claude will orchestrate multiple agents — one might scaffold the project while another sets up the database schema, while a third builds the authentication flow. You'll see progress updates as each agent completes its work.

Stay available You don't need to watch every line of output. But stay around — agents may hit decisions that need your input, like "should this be a modal or a separate page?" Quick answers keep the build moving.
7

Review the output and iterate

Once the agents finish, you have a working codebase. Run it locally following the README, test the flows, and iterate:

You → Claude The login flow works but the dashboard is missing the stats cards from section 3.2 of the requirements. Also, the sidebar should collapse on mobile. Fix both.

Keep iterating until it matches your expectations. Each round gets faster because Claude already has full context of the codebase.

That's the core workflow.

Choosing your tech stack

Tell Claude what to use upfront. Be explicit — if you don't specify, it'll make its own choices (which may not match your preferences). Here's our recommended stack and when to deviate.

→

Frontend: Tailwind CSS + shadcn/ui

Always specify these. Tailwind gives Claude a consistent utility-class system, and shadcn/ui provides a library of well-built, accessible components that Claude knows inside and out. Include this in your prompt:

Add to your prompt "Use Tailwind CSS for styling and shadcn/ui for UI components. Install shadcn components as needed."
→

Framework: Next.js, Node.js, or something else

Next.js (with App Router) is the default choice for full-stack web apps — it handles routing, server-side rendering, and API routes in one package. Use it unless you have a reason not to.

Plain Node.js + Express is better for pure backend services, APIs, or CLIs where you don't need a frontend framework.

React (Vite) works well for SPAs where you want a lighter setup than Next.js and don't need SSR.

Tell Claude which one you want. If you're unsure, say "use Next.js with App Router and TypeScript" and you'll be in good shape for most projects.

→

Database: Supabase (or SQLite for prototypes)

Supabase is our preferred database. It gives you Postgres, authentication, row-level security, and real-time subscriptions out of the box. Tell Claude:

Add to your prompt "Use Supabase for the database and auth. Use the Supabase JS client library. Set up row-level security policies for all tables."

For quick prototypes or internal tools where you don't want to set up Supabase, SQLite (via better-sqlite3 or Drizzle + LibSQL) is a great zero-config alternative. Everything stays in a single file.

Supabase vs. Firebase vs. building from scratch? Supabase wins for us because it's standard Postgres underneath — no vendor lock-in, easy to migrate, and Claude understands SQL well. Firebase is fine but its data model (NoSQL) can be awkward for relational data. Building auth and storage from scratch is almost never worth it.
→

File storage: S3-compatible

For file uploads (images, documents, etc.), use S3-compatible storage. Our preferred options:

  • Supabase Storage — already integrated if you're using Supabase. Easiest path.
  • Hetzner Object Storage — cheap, EU-based, S3-compatible. Good for larger files.
  • AWS S3 — the original. Use if you're already in the AWS ecosystem.

Tell Claude which one to use. The S3 API is the same across all of them, so switching later is straightforward.

→

CMS frameworks: build from scratch vs. PayloadCMS

If your project needs a CMS (blog, content management, admin panel), you have two paths:

Build from scratch with Claude. For simple CRUD admin panels or content management, Claude can build exactly what you need faster than you'd configure a framework. No learning curve, no fighting the framework's opinions, and you get exactly the features you need — nothing more.

Use PayloadCMS (or similar) when you need a full-featured CMS with rich text editing, media management, localization, versioning, and complex content modeling. Prompt Claude like:

Add to your prompt "Use PayloadCMS for the admin panel and content management. Set up collections for [pages, posts, etc.]."
When in doubt, build from scratch With AI-assisted development, the calculus has changed. Frameworks save time when you need 80%+ of their features. If you only need a few CRUD screens and an admin login, building from scratch with Claude is often faster and produces simpler, more maintainable code. You won't spend hours debugging framework-specific issues or working around conventions that don't fit your use case.
→

Always include a README

Every project should have a README.md that explains how to get it running locally. Tell Claude:

Add to your prompt "Include a README.md with: - Prerequisites (Node version, etc.) - How to install dependencies - How to set up environment variables - How to run the dev server - How to run database migrations"

A good README is the difference between a teammate being able to pick up your project in 5 minutes vs. spending an hour guessing at setup steps.

Tips for working with agent teams

→

Use a CLAUDE.md file for persistent instructions

Create a CLAUDE.md file in your project root with conventions Claude should always follow. This gets loaded into every conversation automatically:

CLAUDE.md — example # Project conventions - Use TypeScript strict mode - Use Tailwind CSS + shadcn/ui for all UI - Use Supabase for database and auth - Use Supabase Storage for file uploads - All API routes go in src/app/api/ - Use server actions where possible - Write integration tests for critical flows - Commit messages: imperative mood, no emoji
→

Break big projects into phases

Don't try to build everything in one session. Split the work into phases: "Phase 1: Auth + basic layout. Phase 2: Core CRUD features. Phase 3: Search and filtering." Each phase gets its own plan-review-build cycle.

→

Commit early and often

Tell Claude to commit after each milestone. This gives you checkpoints to roll back to if something goes wrong. A good prompt addition:

Add to your prompt "Create a git commit after completing each task in the plan. Use clear, descriptive commit messages."
→

Review diffs, not just the running app

After agents finish, don't just test the UI — look at the code. Run git diff or use your editor's diff view. Check for hardcoded secrets, unnecessary dependencies, or patterns that don't match your conventions.

→

Be specific about what you don't want

Claude is eager to help and will sometimes add features you didn't ask for. If you want a minimal implementation, say so explicitly:

You → Claude "Build only what's in the plan. No extra features, no premature abstractions, no over-engineering. Keep it simple and working."
→

Use /clear between major context switches

If you're shifting from planning to implementation, or from one feature area to another, start a fresh conversation with /clear. Claude will re-read your project files and CLAUDE.md, giving it a clean mental model without leftover context from the previous discussion.

Useful slash commands

Slash commands start with / and control the tool itself — they're not messages to Claude.

/

/status — Check your usage

See how much of your rate limit you've consumed. If you're running lots of agents, you'll burn through tokens fast.

Hitting rate limits? Agent teams consume tokens quickly. If /status shows you're near your limit, or Claude starts responding slower, ask for help on Slack — we can sort out your usage or give you a different API key.
/

/clear — Start a fresh conversation

Reset the conversation while keeping all project files. Useful between phases or when Claude seems confused by stale context.

/

/help — See all available commands

View the full list of slash commands.

/

/btw — Ask a quick side question

Need a quick answer while agents are working? Type /btw followed by your question. It answers from existing conversation context, shows up as an overlay, and disappears when you press Esc. Doesn't interrupt the main task or add to history.

Common questions

How is this different from Cursor or Copilot?

Cursor and Copilot work at the file level — they help you write code faster in an editor. Claude Code works at the project level — it can scaffold entire applications, run commands, manage dependencies, and dispatch multiple agents to build in parallel. Think of it as a team lead vs. a pair programmer.

Can agents work on the same files without conflicts?

Claude orchestrates the agents to minimize conflicts — it assigns independent tasks to different agents and sequences dependent work. Occasionally there may be minor merge issues, but Claude handles resolving them. If you see something off, just point it out.

How do I handle environment variables and secrets?

Never put secrets in the codebase. Ask Claude to create a .env.example file with placeholder values and add .env to .gitignore. For Supabase, you'll need NEXT_PUBLIC_SUPABASE_URL and NEXT_PUBLIC_SUPABASE_ANON_KEY at minimum — get these from your Supabase project dashboard.

Should I use a CMS framework or build from scratch?

If you need rich text editing, media management, localization, versioning, and complex content workflows — use PayloadCMS or similar. If you need a few admin screens for managing data — build from scratch. With AI-assisted development, simple CRUD is fast to build and simpler to maintain than configuring a framework you'll only use 20% of.

What about testing?

For critical business logic (payments, auth, data processing), ask Claude to write tests. For UI and CRUD, manual testing during development is usually sufficient. A good middle ground: "Write integration tests for the checkout flow and the auth flow. Skip unit tests for basic CRUD operations."

Can I use this for existing projects, not just greenfield?

Yes. Point Claude at your existing codebase and requirements. It will read the existing code, understand the patterns, and propose a plan that works within your existing architecture. Include a CLAUDE.md with your project conventions so it follows your established patterns.

What if the agents produce bad code?

Review the plan before implementation — most bad code comes from bad plans. If the code doesn't meet your standards after implementation, tell Claude what's wrong specifically: "This component is too complex — break it into smaller components" or "Don't use any for TypeScript types — use proper interfaces."

How much does this cost in tokens?

A full plan-review-build cycle for a medium feature (5-10 files) typically costs $2-5 in API tokens. Large builds with many agents can cost more. Use /status to monitor your usage. If you're on a team plan, your API key covers this.
